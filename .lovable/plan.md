

## Fix: Remove Lovable Cloud References — Use External Supabase for Everything

### Problem
The `.env` file is auto-generated by Lovable Cloud and sets `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` to the **Lovable Cloud** project (`ttpndqaghjuoqnngdwlh`). In `src/lib/supabase.ts`, these env vars are checked **before** the hardcoded fallback to your real Supabase (`qrkojbfayjrtrlrmgzry`). So the app ends up connecting to Lovable Cloud instead of your database — which is why customers and materials aren't loading.

Your screenshot confirms all the real data (customers_demo, customers_prod, job_materials, supplier_items, etc.) lives on your external Supabase.

### Fix

**File: `src/lib/supabase.ts`**

Remove the `VITE_SUPABASE_URL` / `VITE_SUPABASE_PUBLISHABLE_KEY` env var checks entirely. These will always resolve to Lovable Cloud values (since the `.env` is auto-managed). Keep only the `VITE_EXT_*` overrides and the hardcoded fallback to your real project:

```
const SUPABASE_URL =
  import.meta.env.VITE_EXT_SUPABASE_URL ||
  'https://qrkojbfayjrtrlrmgzry.supabase.co';

const SUPABASE_ANON_KEY =
  import.meta.env.VITE_EXT_SUPABASE_ANON_KEY ||
  'eyJ...real key...4oA';
```

This ensures **both demo and prod** always hit your external Supabase, regardless of what Lovable puts in `.env`.

### What this affects
- Demo mode: reads from `customers_demo`, `demo_session_jobs`, etc. on your Supabase
- Prod mode: reads from `customers_prod`, `job_materials`, `supplier_items`, etc. on your Supabase
- Auth: login/signup checks against your Supabase auth
- No code changes needed anywhere else — all services already import from `src/lib/supabase.ts`
